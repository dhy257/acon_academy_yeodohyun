# 01. 인터페이스와 라이브러리 설계

## 학습 목표

- 라이브러리를 만들 때 인터페이스가 왜 필요한지 이해한다.
- 인터페이스가 "기능의 약속"이라는 점을 설명할 수 있다.
- 인터페이스의 `default` 매서드와 `static` 매서드를 구분할 수 있다.

## 1. 라이브러리와 인터페이스의 관계

라이브러리는 다른 사람이 가져다 쓸 수 있도록 미리 만들어 둔 코드 묶음이다. 예를 들어 `Arrays.sort()`, `Collections.sort()` 같은 기능은 우리가 정렬 알고리즘을 직접 만들지 않아도 정렬을 할 수 있게 해 준다.

하지만 라이브러리를 만든 사람은 우리가 나중에 어떤 객체를 만들지 알 수 없다.

예를 들어 `Acorn`, `Student`, `Customer`, `Score` 같은 클래스는 우리가 직접 만든다. 라이브러리 제작자는 이런 클래스 안에 `name`, `age`, `score` 중 무엇이 있는지 알 수 없다. 그래서 라이브러리는 이렇게 말한다.

> 정렬 기능은 내가 제공할게. 대신 너는 정렬 기준을 약속된 방식으로 알려줘.

이때 사용하는 약속이 인터페이스이다.

## 2. 인터페이스는 기능의 약속이다

인터페이스는 "이 기능을 사용하려면 이 매서드를 반드시 만들어야 한다"라는 규칙이다.

```java
interface MyComparable {
    int compareTo(Object other);
}
```

이 인터페이스를 구현하는 클래스는 반드시 `compareTo()`를 만들어야 한다.

```java
class Score implements MyComparable {
    String name;
    int kor;

    Score(String name, int kor) {
        this.name = name;
        this.kor = kor;
    }

    @Override
    public int compareTo(Object other) {
        Score s = (Score) other;
        return this.kor - s.kor;
    }
}
```

`return this.kor - s.kor;`의 의미는 다음과 같다.

- 양수: 현재 객체가 더 크다.
- 0: 두 객체가 같다.
- 음수: 현재 객체가 더 작다.

정렬 라이브러리는 이 결과를 보고 두 객체의 순서를 바꿀지 결정한다.

## 3. 라이브러리 입장에서 생각하기

아래는 정렬 라이브러리를 아주 단순하게 흉내 낸 코드이다.

```java
class MyArrays {
    public static void sort(Object[] list) {
        for (int i = 0; i < list.length - 1; i++) {
            for (int j = i + 1; j < list.length; j++) {
                MyComparable left = (MyComparable) list[i];

                if (left.compareTo(list[j]) > 0) {
                    Object temp = list[i];
                    list[i] = list[j];
                    list[j] = temp;
                }
            }
        }
    }
}
```

여기서 `MyArrays`는 `Score`의 필드가 무엇인지 모른다. 대신 `MyComparable`이라는 약속만 믿는다.

즉, 라이브러리 설계자는 정렬의 흐름을 만들고, 라이브러리 사용자는 정렬 기준을 구현한다.

## 4. Comparable 방식과 Comparator 방식의 차이

정렬 기준을 제공하는 대표적인 방식은 두 가지이다.

### Comparable

객체 자신이 기본 정렬 기준을 가진다.

```java
class Acorn implements Comparable<Acorn> {
    String name;
    int score;

    @Override
    public int compareTo(Acorn other) {
        return this.score - other.score;
    }
}
```

`Acorn` 객체는 기본적으로 점수 오름차순 정렬 기준을 갖는다.

### Comparator

객체 밖에서 정렬 기준을 따로 제공한다.

```java
Comparator<Acorn> byName = new Comparator<Acorn>() {
    @Override
    public int compare(Acorn a1, Acorn a2) {
        return a1.name.compareTo(a2.name);
    }
};
```

`Comparator`는 상황에 따라 여러 정렬 기준을 만들 때 좋다.

## 5. 인터페이스의 default 매서드

원래 인터페이스는 추상 매서드만 가진다고 배웠을 수 있다. 하지만 Java 8부터는 인터페이스 안에 기본 구현을 가진 `default` 매서드를 만들 수 있다.

```java
interface Cookable {
    void cook();

    default void prepare() {
        System.out.println("재료를 준비합니다.");
    }
}
```

구현 클래스는 `cook()`은 반드시 구현해야 하지만, `prepare()`는 그대로 써도 되고 필요하면 오버라이드할 수 있다.

```java
class KoreanCook implements Cookable {
    @Override
    public void cook() {
        System.out.println("김치찌개를 끓입니다.");
    }
}
```

`default` 매서드는 기존 구현 클래스를 깨뜨리지 않고 인터페이스에 기능을 추가할 때 유용하다.

## 6. 인터페이스의 static 매서드

인터페이스의 `static` 매서드는 객체를 만들지 않고 인터페이스 이름으로 바로 호출한다.

```java
interface Cookable {
    static void printRule() {
        System.out.println("요리 전 손을 씻습니다.");
    }
}
```

호출 방법은 다음과 같다.

```java
Cookable.printRule();
```

`static` 매서드는 구현 클래스가 오버라이드하는 기능이 아니다. 인터페이스에 붙어 있는 공용 도구 매서드라고 생각하면 된다.

## 7. default와 static 비교

| 구분 | default 매서드 | static 매서드 |
|---|---|---|
| 호출 방법 | 구현 객체로 호출 | 인터페이스 이름으로 호출 |
| 오버라이드 | 가능 | 불가능 |
| 목적 | 구현 클래스에 기본 기능 제공 | 인터페이스 관련 공용 기능 제공 |
| 예 | `cook.prepare()` | `Cookable.printRule()` |

## 8. 정리

- 인터페이스는 라이브러리와 사용자가 지켜야 할 약속이다.
- 라이브러리는 전체 알고리즘을 제공하고, 사용자는 인터페이스를 구현해서 기준을 제공한다.
- `Comparable`은 객체 안에 기본 정렬 기준을 넣는 방식이다.
- `Comparator`는 객체 밖에서 정렬 기준을 따로 제공하는 방식이다.
- `default` 매서드는 구현 클래스가 물려받아 사용할 수 있고 오버라이드도 가능하다.
- `static` 매서드는 인터페이스 이름으로 직접 호출하는 공용 기능이다.

3. `Printable`에 `default void preview()`를 추가하고 실행해 보자.
4. `Printable`에 `static void printGuide()`를 추가하고 `Printable.printGuide()`로 호출해 보자.
5. 왜 라이브러리 제작자가 인터페이스를 사용해야 하는지 한 문장으로 설명해 보자.
