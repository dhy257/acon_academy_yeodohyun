안녕하세요, 오늘은 객체지향 프로그래밍, OOP에 대해 간단히 소개하겠습니다.

먼저, 객체지향의 핵심 개념 중 하나는 **캡슐화**입니다.
캡슐화란 데이터와 데이터를 처리하는 메서드를 하나로 묶는 것을 말합니다.
즉, 외부에서는 데이터를 직접 접근할 수 없고, 메서드를 통해서만 데이터를 다룰 수 있습니다.
이렇게 함으로써 잘못된 값 입력을 막아 데이터 보호가 가능하고,
코드 유지보수도 쉬워지며, 모듈화로 독립적인 단위 관리가 가능합니다.

예를 들어, Person 클래스에서 이름(String name)을 관리한다고 할 때,
private로 변수를 숨기고, setName, getName 메서드로만 접근하게 만들 수 있습니다.
이렇게 하면 잘못된 값이 들어가는 것을 방지할 수 있습니다.

다음은 절차지향과 객체지향의 차이입니다.
절차지향은 데이터와 함수를 분리해서 사용합니다. 
하지만 코드가 복잡해지고, 데이터 보호가 어렵다는 단점이 있습니다.
반면, 객체지향은 데이터와 메서드를 함께 묶어서 객체를 만들기 때문에
코드 관리가 쉽고 구조가 명확하며 데이터 보호도 가능합니다.

그럼 클래스와 객체에 대해 간단히 설명하겠습니다.
클래스는 객체를 만들기 위한 설계도이고,
객체는 그 설계도를 바탕으로 실제 메모리에 생성된 인스턴스입니다.
예를 들어, Student라는 클래스를 만들고, 
Student s = new Student(); 처럼 new를 통해 객체가 생성합니다.

다음으로 **this와 생성자**입니다.
this는 현재 객체 자신을 가리키는 참조 변수입니다.
객체 변수와 매개변수를 구분할 때 사용하며,
static 메서드에서는 사용할 수 없습니다.
생성자는 객체를 만들 때 자동으로 호출되어 객체 변수를 초기화합니다.
생성자가 없으면 컴파일러가 기본 생성자를 만들어주고,
필요하면 매개변수를 받는 생성자를 정의해 초기값을 설정할 수 있습니다.

그리고 **접근제어자와 Getter/Setter**입니다.
접근제어자는 private, default, protected, public이 있으며,
클래스와 패키지, 상속 관계에 따라 접근 범위를 설정합니다.
Getter와 Setter는 캡슐화된 변수를 안전하게 읽고 수정할 수 있는 메서드입니다.

마지막으로 **static**입니다.
static은 객체를 만들지 않아도 클래스에 속한 변수나 메서드를 사용할 수 있게 합니다.
모든 객체가 공유하며, static 메서드에서는 this를 사용할 수 없습니다.

정리하면, 캡슐화를 통해 데이터 보호와 코드 관리가 용이해지고,
this와 생성자로 객체를 초기화하며,
접근제어자와 Getter/Setter로 안전하게 데이터를 다루고,
static으로 객체 없이 클래스 멤버를 사용할 수 있습니다.
